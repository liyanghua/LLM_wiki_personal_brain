# 《专家经验 → SOP → 流程图 → 本体表 → 决策表 → Agent Harness → Eval 闭环》实施蓝图

## 0. 一句话定义

这条实施蓝图的目标，不是把专家经验沉淀成一堆 SOP 文档，而是把专家的隐性 know-how，持续编译为：

- 可解释的**本体资产**
- 可复用的**决策单元**
- 可执行的**Agent Harness**
- 可验证的**评测闭环**

最终形成一个从单场景起步、可逐步扩展到整个经营过程的 **本体大脑资产生产线**。

---

## 1. 总体蓝图：七段流水线

```text
专家经验
  ↓
SOP
  ↓
流程图
  ↓
本体表
  ↓
决策表
  ↓
Agent Harness
  ↓
Eval 闭环
```

更完整地说，是下面这条链：

```text
场景定义
  → 专家访谈 / 问卷 Agent
  → SOP 迭代
  → 流程图外化
  → 本体抽取
  → 决策单元结构化
  → Agent 执行编排
  → 评测 / 反馈 / 复盘
  → 反哺 SOP / 本体 / 决策 / Harness
```

这 7 段不是串行一次性工程，而是一个持续进化的闭环。

---

## 2. 七段流水线的实施定义

## 阶段 1：专家经验采掘

### 目标
把“专家脑子里的隐性经验”从模糊经验，转成可追问、可比较、可沉淀的原始材料。

### 输入
- 场景问题定义
- 初版 SOP
- 样例案例
- 业务目标与约束

### 方法
- 递进式问卷 Agent
- 对比式访谈
- 案例复盘式访谈
- think-aloud 现场操盘访谈

### 产物
- 专家原始问答
- 边界条件
- 常见误区
- 例外情况
- 决策理由
- 反例知识

### 关键要求
- 不只问“怎么做”，还要问“为什么这么做”
- 不只问正例，还要问“什么时候不要这么做”
- 对同一问题至少收集 2–3 位专家视角，便于后续冲突消解

### 这一阶段的风险
- 口述知识偏个人风格
- 术语不统一
- 专家会省略默认前提
- 回答容易停留在表层步骤

### 控制办法
- 强制追问：触发条件 / 反例 / 边界 / 风险 / 替代方案
- 同问题跨专家对比
- 给案例而不是只问抽象问题

---

## 阶段 2：SOP 编译

### 目标
把专家原始经验整理为“人能理解、能讨论、能修改”的场景方法学版本。

### 输入
- 专家访谈记录
- 问卷 Agent 输出
- 真实案例拆解

### 产物
- SOP V1 / V2 / V3
- 步骤说明
- 判断分支
- 注意事项
- 反例与常见误区

### SOP 的正确定位
SOP 不是最终本体，也不是最终执行逻辑，而是：

- 专家经验的第一层外化载体
- 本体和决策单元的上游来源
- 供专家审阅和共创的中间层

### SOP 模板建议
每个 SOP 节点至少包括：
- 节点名称
- 节点目的
- 输入
- 输出
- 关键判断
- 依赖证据
- 常见错误
- 例外处理
- 是否需人工确认

### 风险
- SOP 过度流程化，缺少对象与状态抽象
- SOP 文档越写越长，但复用性越来越差

### 控制办法
- 强制每个节点都写清输入 / 输出 / 判断 / 证据
- 每版 SOP 评审时增加“能否抽成对象、状态、动作”检查

---

## 阶段 3：流程图外化

### 目标
把 SOP 中的执行顺序、判断分支和协作节点，外化成可视化流程。

### 输入
- SOP 文档
- 场景边界说明

### 产物
- 流程图
- 状态分支图
- 人机协同图
- 审批与回写节点图

### 流程图的正确定位
流程图不是终点，而是“本体抽取源”和“Agent 编排源”。

### 从流程图中必须抽出的 6 类信息
每一个流程节点，都要强制抽取：

1. 对象：这个节点处理什么对象
2. 状态：这个节点在判断什么状态
3. 证据：这个节点依赖什么证据
4. 动作：这个节点执行什么动作
5. 约束：这个节点受什么约束
6. 输出：这个节点产出什么结构化对象

### 风险
- 流程图只保留顺序，不保留语义
- 流程图适合展示，但不适合直接执行和评测

### 控制办法
- 流程图必须配套“节点结构化表”
- 每个节点都有唯一 ID，便于映射到本体与决策表

---

## 阶段 4：本体表构建

### 目标
从 SOP 和流程图中抽出稳定的业务世界模型。

### 本体表的核心任务
不是描述“先做什么”，而是描述：
- 这里有什么对象
- 对象处于什么状态
- 对象之间有什么关系
- 什么证据可以支撑状态判断

### 本体表至少拆成 6 张

#### 4.1 对象表
字段建议：
- object_id
- object_name
- 定义
- 类型
- 核心属性
- 上下游关系
- 适用场景
- 数据来源

#### 4.2 状态表
字段建议：
- state_id
- state_name
- 所属对象
- 定义
- 判定条件
- 证据要求
- 常见误判
- 相关风险

#### 4.3 关系表
字段建议：
- relation_id
- source_object
- target_object
- relation_type
- 生效条件
- 强弱程度
- 示例

#### 4.4 证据表
字段建议：
- evidence_id
- evidence_type
- 来源
- 支撑对象/状态
- 可信度
- 更新频率
- 缺失处理

#### 4.5 约束表
字段建议：
- constraint_id
- constraint_name
- 类型（品牌/平台/资源/时间/风险）
- 生效范围
- 优先级
- 冲突处理方式

#### 4.6 动作表
字段建议：
- action_id
- action_name
- 作用对象
- 触发状态
- 前置条件
- 风险
- 预期效果
- 可执行性要求

### 风险
- 对象粒度过细，维护成本爆炸
- 状态定义过泛，无法指导决策
- 专家术语无法统一

### 控制办法
- 先做“场景最小本体”，不要一开始追求全域
- 术语统一成语义字典
- 每个状态都要求最少一个证据锚点

---

## 阶段 5：决策表构建

### 目标
把“业务世界模型”升级成“可运行的决策单元”。

### 决策表是本体大脑最关键的桥梁
本体表回答“世界是什么样”；
决策表回答“在这种世界状态下，应该怎么办”。

### 决策单元最小 Schema
每条决策记录至少包括：
- decision_id
- 决策名称
- 适用场景
- 输入对象
- 触发状态
- 所需证据
- 判断逻辑
- 候选动作
- 动作优先级
- 风险提示
- 是否需审批
- 验证指标
- 适用边界
- 反例模式

### 决策表建议拆成 4 类

#### 5.1 判断规则表
- 什么证据组合会触发什么判断

#### 5.2 动作选择表
- 在某种状态下，哪些动作优先

#### 5.3 风险边界表
- 哪些情况必须人工复核或禁止自动推进

#### 5.4 验证指标表
- 如何判断这条决策是否有效

### 风险
- 决策规则写成模糊自然语言，无法执行
- 缺失反例边界，Agent 容易误判

### 控制办法
- 决策表要结构化，不写长段散文
- 每条决策必须绑定证据、动作、验证指标
- 加入“禁行条件 / 审批条件”字段

---

## 阶段 6：Agent Harness 落地

### 目标
让本体表和决策表真正进入运行态，而不是停留在知识层。

### Agent Harness 的核心组成

#### 6.1 认知 Harness
负责：
- 输入解析
- 对象识别
- 状态识别
- 证据收集
- 术语规范化

输入：原始内容 / 数据 / 文档 / 任务上下文  
输出：结构化对象、状态与证据包

#### 6.2 决策 Harness
负责：
- 调用决策表
- 匹配触发状态
- 生成候选方案
- 应用约束与风险规则

输入：对象 / 状态 / 证据 / 目标 / 约束  
输出：Decision Object / Opportunity / Brief / Strategy Pack

#### 6.3 执行 Harness
负责：
- 把决策结果转成任务单
- 调用生成工具 / 发布工具 / 审批流
- 记录执行状态
- 写回资产

输入：决策对象  
输出：执行结果 / 发布记录 / 资产回写

#### 6.4 学习 Harness
负责：
- 收集评分、反馈、结果
- 识别 bad case
- 触发规则升级
- 写回案例与测试集

输入：执行结果 / 人工反馈 / 线上效果  
输出：新规则 / 新样例 / 新 guardrail / Skill 升级建议

### 你们当前模块如何挂进去
- Gateway：入口与调度
- Context Compiler：上下文编译
- Data Agent：结构化证据供给
- RAG Agent：案例与非结构化证据供给
- Memory：历史经验和失败模式
- Eval：结果校验与评分
- Ontology Brain：对象语义、状态、动作和评测口径内核

### 风险
- 本体和决策表存在，但 Agent 仍主要靠 prompt 临场发挥
- 缺少执行前 / 执行后护栏

### 控制办法
- Agent 只消费结构化对象和决策表，不直接吞长文 SOP
- 每步输出都需要 schema 校验和 rule check
- 明确哪些节点必须人工审批

---

## 阶段 7：Eval 闭环

### 目标
让整套系统不是“能跑”，而是“越跑越准”。

### Eval 必须覆盖三层

#### 7.1 输入质量评测
- 输入是否完整
- 关键证据是否缺失
- 对象识别是否正确
- 状态判定是否稳定

#### 7.2 决策质量评测
- 证据是否充分
- 判断是否合理
- 动作是否可执行
- 是否违反约束
- 是否命中反例模式

#### 7.3 结果效果评测
- 人工评分
- 可行动性
- 业务效果
- 风险控制情况
- 资产沉淀质量

### 评测表字段建议
- task_id
- scene_type
- input_completeness_score
- evidence_sufficiency_score
- decision_quality_score
- action_feasibility_score
- risk_control_score
- human_overall_score
- business_effect_score
- fail_type
- remediation_action

### Bad case 体系
每个失败样本都要归入：
- 认知错
- 决策错
- 执行错
- 评测错
- 数据/上下文错

并决定：
- 是否加入测试集
- 是否升级 guardrail
- 是否修订本体
- 是否修订决策表

## 3. 七段流水线的角色分工

| 阶段 | 主要角色 | 核心职责 |
|---|---|---|
| 专家经验采掘 | 行业专家、访谈 PM、问卷 Agent 设计者 | 榨取 tacit knowledge、边界条件、反例 |
| SOP 编译 | 业务专家、产品经理 | 把经验整理成可审阅的方法学结构 |
| 流程图外化 | 产品经理、流程设计师 | 外化步骤、分支、审批与回写节点 |
| 本体表构建 | 本体负责人、数据/算法 | 抽对象、状态、关系、证据、约束、动作 |
| 决策表构建 | 本体负责人、策略专家、算法 | 抽取决策单元与动作逻辑 |
| Agent Harness 落地 | 智能体平台、后端、算法 | 让本体和决策进入运行态 |
| Eval 闭环 | 评测负责人、业务专家、算法 | 建测试集、评分机制、失败闭环 |

## 4. 每个阶段的验收标准

## 阶段 1 验收
- 至少沉淀出 20–30 条高价值专家判断
- 包含边界、反例、例外，而不只是主流程
- 至少完成 2 位专家的冲突对比

## 阶段 2 验收
- SOP 节点具备输入 / 输出 / 判断 / 证据 / 风险字段
- 专家可读、可修改、可评审

## 阶段 3 验收
- 流程图每个节点都有唯一 ID
- 每个节点都能映射到对象 / 状态 / 动作 / 输出

## 阶段 4 验收
- 有最小可用本体表
- 关键对象与状态定义明确
- 证据与状态有映射关系

## 阶段 5 验收
- 至少形成 20–50 条结构化决策单元
- 每条决策都绑定证据、动作、验证指标、风险边界

## 阶段 6 验收
- Agent 不再直接依赖长文 SOP
- 能消费结构化本体与决策对象
- 关键节点有 rule check 与人工审批插点

## 阶段 7 验收
- 有稳定评测表与人工评分机制
- 有 bad case 分类与回流机制
- 每周可产出规则升级 / 资产升级列表

---

## 5. 推荐的最小落地顺序

### 第一步：单场景试点
先选一个场景，例如“生图策划”。
目标不是覆盖全流程，而是跑通全闭环。

### 第二步：做四件套
围绕单场景，一次性补齐：
- SOP
- 流程图
- 本体表
- 决策表

### 第三步：接一个最小 Agent Harness
让 Agent 消费结构化资产，输出一个标准结果对象。

### 第四步：接人工评测与 bad case 回流
先把学习闭环跑通。

### 第五步：再复制到第二个场景
例如从“生图策划”扩到“内容选题”或“机会卡生成”。

这样才能验证你们抽的本体骨架是否真能跨场景复用。

---

## 6. 你们现在最应该补的不是更多访谈，而是三张关键表

### 表 1：本体对象表
把流程图里的对象、状态、证据、动作抽出来。

### 表 2：决策单元表
把“什么时候判断什么、做什么动作、如何验证”结构化出来。

### 表 3：评测与 bad case 表
把“做得好不好、为什么错、怎么改”闭环起来。

这三张表一补，你们的路线就会从“专家经验采掘系统”升级成“本体大脑资产生产线”。

---

## 7. 一句总结

**真正的目标不是把专家经验写成更完整的 SOP，而是把专家经验编译为：可定义对象、可表达状态、可绑定证据、可生成决策、可执行动作、可评测进化的本体大脑运行资产。**

也可以更简洁地说：

**从专家说法，升级到系统可运行的决策资产。**
